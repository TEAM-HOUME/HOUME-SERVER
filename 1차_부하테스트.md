# 1차 부하테스트 운영 기록

## 1. 목적
- Gemini 이미지 생성 API 부하테스트(1차)를 안정적으로 실행하기 위한 운영 체크포인트를 정리한다.
- 코드로 반영 가능한 항목은 별도 반영했고, 이 문서는 운영 환경에서 추가로 인지해야 할 사항만 다룬다.

## 2. 코드 반영 완료 항목 (참고)
- `k6/lib/common.js`
  - 객체 스프레드(`...extraTags`)를 `Object.assign`으로 변경해 런타임 호환성 확보
  - `REQUEST_TIMEOUT` 환경변수 추가(기본 `220s`)
- `k6/seeds/load_test_seed.sql`
  - 부하테스트 최소 데이터(유저/집/도면/무드보드/태그/크레딧) 시드 SQL 추가
- `k6/scripts/seed_load_test_data.sh`
  - DB 환경변수 기반 시드 실행 스크립트 추가
- `k6/README.md`
  - 시드 절차 및 `REQUEST_TIMEOUT` 가이드 추가

## 3. 운영에서 인지해야 하는 이슈

### 3.1 SSH/Pem 권한
- PEM 권한이 `0644`면 SSH 접속이 거부된다.
- 권한은 반드시 `chmod 600 <pem>`으로 맞춰야 한다.

### 3.2 DB 네트워크 접근
- 테스트 서버에서 RDS `5432` 접근이 막히면 시드/애플리케이션 모두 실패한다.
- 보안그룹 인바운드(소스 IP)와 VPC 라우팅을 먼저 확인해야 한다.

### 3.3 메모리/스왑
- 저메모리 인스턴스에서 도커 빌드/부팅 시 스왑 의존도가 높아져 전체 시간이 크게 늘어난다.
- 부하테스트 전 최소 체크:
  - `free -h`
  - `swapon --show`
  - JVM/Gradle 메모리 옵션 과도 설정 여부

### 3.4 Spring 프로파일 충돌
- `spring.profiles.include`를 profile-specific 파일에 잘못 넣으면 부팅 루프가 발생할 수 있다.
- `load_test` 프로파일의 목적(Stub 사용, 외부 API 비활성)을 명확히 분리해 유지한다.

### 3.5 테스트 데이터 공백
- RDS가 비어 있으면 API는 권한/존재성 검증에서 즉시 실패한다.
- 시나리오 시작 전 아래는 반드시 존재해야 한다.
  - 대상 `houseId`
  - 요청에 맞는 `floorPlanId`, `moodBoardId`
  - 테스트 유저의 ACTIVE 크레딧

### 3.6 토큰/락 경합
- 단일 유저 토큰으로 동시 요청 시 `credit_lock_user_{id}` 락 경합으로 실패율이 급증한다.
- 동시성 테스트에서는 다중 유저 토큰 풀을 사용해야 한다.

### 3.7 실행 시간 기대치
- Stub 지연(90~120초) 기준으로 `smoke -> ramp -> stress -> soak`는 3시간 이상 소요될 수 있다.
- 중간 실패를 방지하려면 장시간 세션 유지(터미널, SSH) 전략이 필요하다.

## 4. 부하테스트 중 k6 모니터링 방법
- 가능하다. 아래 방법으로 실시간 모니터링할 수 있다.

### 4.1 기본 CLI 실시간 출력
```bash
k6 run k6/scenarios/smoke.js
```
- 콘솔에서 VU, iterations, 실패율, p95를 실시간으로 본다.

### 4.2 백그라운드 실행 로그 모니터링
```bash
tail -f /home/ubuntu/loadtest/logs/nohup_k6_sequence_*.out
```
- 서버에서 `nohup`으로 실행한 시퀀스 진행상황을 추적한다.

### 4.3 k6 Web Dashboard
```bash
K6_WEB_DASHBOARD=true \
K6_WEB_DASHBOARD_PORT=5665 \
k6 run k6/scenarios/smoke.js
```
- 로컬 브라우저에서 대시보드로 실시간 지표 확인 가능.

### 4.4 외부 시계열 모니터링 (선택)
- Prometheus/Grafana를 쓰면 장시간 테스트 비교가 쉽다.
- 운영팀 공용 대시보드를 쓰는 경우, 시나리오 태그(`scenario`) 기준으로 분리 조회 권장.
