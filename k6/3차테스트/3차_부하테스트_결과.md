## 3. 테스트 결과 판단
- 결론: 3차 테스트는 **저부하 구간에서 성공률/응답시간 모두 기준을 충족**했다.
- 근거 1: `http_req_failed = 0%`, `checks = 100%`로 HTTP/API 실패는 없었다.
- 근거 2: `http_req_duration p95 = 24.35s`, `p99 = 24.49s`로 시나리오 임계치 `p(95) < 30s`, `p(99) < 45s`를 모두 만족했다.
- 근거 3: 18분 동안 `315건`을 처리했고, 모든 iteration이 정상 완료됐다.

## 4. 2차 대비 해석
1. 2차 테스트의 `p95 149.24s`가 3차에서 `24.35s`로 크게 감소했다.
2. 총 처리 요청 수도 `109건 -> 315건`으로 증가했다.
3. 즉 `60초 stub` 환경에서 보였던 긴 대기와 적체는, 긴 외부 처리시간이 큐에 누적되면서 증폭된 영향이 컸다고 볼 수 있다.
4. 반대로 `10초 stub` 환경의 `1~10 VU` 저부하 구간에서는 애플리케이션이 안정적으로 요청을 소화했다.

## 5. 현재 시점의 판단
1. `1~10 VU` 범위, `10초 stub` 조건에서는 현재 구조가 충분히 안정적이다.
2. 다만 평균 응답시간이 `15.55초`인 만큼, stub 10초 대비 애플리케이션 오버헤드는 여전히 약 `5~15초` 수준 존재한다.
3. 따라서 다음 단계는 "통과 여부"보다 `10초 -> 30초 -> 60초`로 hold를 늘렸을 때 어느 지점에서 지연이 급증하는지 찾는 쪽이 더 의미 있다.

## 6. 해석 시 주의사항
1. 이 결과는 `load_test` 프로필의 `Gemini Stub` 기준이다.
2. 실제 Gemini 외부 API 호출 성능이나 비용 구조와 동일한 결과로 해석하면 안 된다.
3. 이번 3차 테스트는 "앱 자체의 저부하 처리 안정성"을 보는 용도로 사용하는 게 맞다.
