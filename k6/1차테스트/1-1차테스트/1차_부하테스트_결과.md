## 3. 테스트 결과 판단
- 결론: 현재 부하 구간은 **정상 처리 구간이 아니라 병목/장애 구간**입니다.
- 근거 1: 실패율이 98%대라서 트래픽 처리 안정성이 없음
- 근거 2: p95가 145초로 목표 응답시간(500ms)과 큰 차이
- 근거 3: 이전 로그에서 `504 upstream timed out`, `Hikari pool timeout`이 동반됨
- 근거 4: `v3` 경로는 이미지 2장을 비동기 생성하고 `orTimeout(200s)` 대기 구조라 대기열이 누적되기 쉬움

## 4. 병목 원인(현재 가설)
1. Stub 자체 지연(90~120초)으로 장시간 요청이 기본값으로 발생
2. 비동기 풀(`core 4/max 8/queue 20`) 포화 시 대기열 증가
3. DB 커넥션 대기(Hikari timeout) + Nginx upstream timeout으로 실패 전이
4. 토큰/크레딧 락 경합이 실패율을 추가로 악화시킬 가능성

## 5. 즉시 후속 액션
1. 현 런 종료 후 `--summary-export`로 최종 JSON 보관
2. 동일 환경에서 저부하 스텝 테스트(1,2,4,6,8,10 VU)로 임계점 재측정
3. 테스트 토큰을 다중 사용자 풀로 분산해서 락 경합 변수 제거
4. 병목 분리 측정: (A) Stub 지연 유지, (B) Stub 지연 5~10초 축소 두 케이스 비교
