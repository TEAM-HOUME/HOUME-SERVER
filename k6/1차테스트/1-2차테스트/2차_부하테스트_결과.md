## 3. 테스트 결과 판단
- 결론: 2차 테스트는 **성공률 문제는 해소했지만, 응답시간 기준으로는 실패**했다.
- 근거 1: `http_req_failed = 0%`, `checks = 100%`로 HTTP/API 실패는 없었다.
- 근거 2: 하지만 `http_req_duration p95 = 149.24s`라서 시나리오 임계치 `p(95) < 90s`, `p(99) < 120s`를 만족하지 못했다.
- 근거 3: k6 CLI summary에서도 `thresholds on metrics 'http_req_duration' have been crossed`가 출력되며 컨테이너가 종료 코드 `99`로 종료됐다.
- 근거 4: Stub 지연을 `60초`로 낮췄는데도 평균이 `92초`, p95가 `149초`까지 올라간 것은 처리 대기열이 누적됐다는 뜻이다.

## 4. 1차 대비 해석
1. 1차 테스트에서 보였던 대량 실패(`http_req_failed` 98%대)는 사라졌다.
2. 다중 토큰 분산으로 크레딧 락 경합 변수는 크게 줄어든 것으로 보인다.
3. 다만 응답시간은 여전히 Stub 고정시간(`60초`)보다 훨씬 길어, 애플리케이션 내부 대기나 비동기 작업 큐 적체가 남아 있다.
4. 즉 2차 테스트는 "장애 구간"에서 "무실패지만 느린 구간"으로 이동한 결과다.

## 5. 현재 시점의 판단
1. `1~10 VU` 범위에서는 서버가 요청을 끝까지 처리할 수는 있다.
2. 하지만 "사용 가능한 성능"으로 보기는 어렵다. 이유는 `p95`가 2분 29초 수준이기 때문이다.
3. 따라서 다음 단계는 실패율이 아니라 **응답시간 병목 분리**에 초점을 맞춰야 한다.

## 6. 다음 테스트 제안

2. 같은 시나리오로 `1 -> 2 -> 4 -> 6 -> 8 -> 10 VU`를 유지하고, `p95`가 언제부터 급격히 증가하는지 다시 측정한다.
