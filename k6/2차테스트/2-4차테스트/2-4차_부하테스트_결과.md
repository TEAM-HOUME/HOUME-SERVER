## 4. 해석
1. 같은 `30 VU` 시나리오인데도 이번 런은 전 구간에서 실패가 없었다.
2. `2-3`에서 보였던 `DB pending -> 500` 패턴이 사라졌고, 응답시간도 `60초 stub`에 거의 밀착했다.
3. `p95`가 `132.05s -> 60.26s`로 내려간 건, 이전 런에서 존재했던 추가 대기 구간이 거의 사라졌다는 뜻이다.
4. 최대 응답시간도 `81.01s` 수준으로 제한됐다. 즉 일부 요청에서만 소폭의 후처리 지연이 있었고, 장시간 tail latency는 재현되지 않았다.

## 6. OSIV 관점 해석
1. 이번 배포 후 앱 로그에서 `spring.jpa.open-in-view is enabled by default` 경고는 더 이상 관측되지 않았다.
2. 동시에 `Hikari pending max = 0`, `hikaricp_connections_timeout_total = 0`, `500 count = 0`이 확인됐다.
3. 따라서 `OSIV on` 상태에서 컨트롤러 응답 완료 시점까지 세션/커넥션이 길게 유지되던 가능성은 실제 병목 후보였다고 볼 수 있다.
4. 다만 이번 개선은 `OSIV off` 하나만의 결과가 아니다. 아래 세 변화가 함께 반영됐다.
   - `Hikari max pool size 20`
   - `virtual concurrency limit 32`
   - `/access?userId=`를 통한 멀티 유저 토큰 발급 정상화
5. 따라서 현재 단계의 결론은 `OSIV off가 유의미한 후보였고, 적어도 현재 조합에서 문제를 악화시키지 않았으며 결과는 확실히 좋아졌다`까지가 정확하다.

## 7. 직전 비정상 런과의 구분
- `/tmp/step_mid_load_v4_pool20_20260308_213640.json` 런은 `failed_rate=94.78%`, `median=52ms`였다.
- 이 값은 성능 개선이 아니라, 잘못된 런타임 이미지/토큰 상태에서 다수 요청이 즉시 실패해 왜곡된 결과였다.
- 이번 `2-4`는 새 이미지를 다시 빌드/배포하고, `user1~200` 토큰을 실제 서버에서 재발급한 뒤 얻은 첫 정상 런이다.

## 8. 결론
1. `30 VU` 시나리오에서 이번 조합은 안정적으로 통과했다.
2. `2-3`의 주병목이던 `Hikari pending`과 DB timeout은 재현되지 않았다.
3. 지금 기준으로는 `30 VU`가 더 이상 즉시 실패 구간이 아니다. 다음 임계점 탐색은 `35~50 VU` 구간으로 올려도 된다.
4. 다만 이번 결과를 `OSIV off 단독 승리`로 보면 안 된다. 다음 phase에서는 구조 리팩터링 전이라도, 현재 조합을 baseline으로 두고 `35/40/45/50 VU` 확장 테스트를 해보는 게 맞다.
