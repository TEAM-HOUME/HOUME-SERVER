## 4. 해석
1. 큰 이점은 분명하다. 동일한 `60초` Stub 조건에서 `p95`가 `149.24s -> 75.45s`로 거의 절반 수준까지 줄었다.
2. 총 처리 요청도 `109 -> 165`로 늘어 동일 시간 동안 더 많은 요청을 소화했다.
3. 즉 이번 비동기 스레드풀 확장은 `큐 적체 완화`에 실제 효과가 있었다고 볼 수 있다.
4. 평균과 p95가 모두 `60초` Stub 지연에 가까워졌다는 점이 중요하다. 이전에는 Stub 자체보다 애플리케이션 내부 대기시간이 훨씬 더 컸지만, 이번에는 그 추가 대기가 크게 줄었다.

2. 하지만 종료 직후 서버 로그를 확인한 결과, 이 실패는 자원 고갈보다 `load_test` Stub의 실패 시뮬레이션 영향으로 보는 게 맞다.
3. 근거:
   - `/Users/jeonjaeyeon/Desktop/houme/HOUME-SERVER/src/main/resources/application-load_test.yml` 에서 `load-test.gemini.failure-rate` 기본값이 `0.02`다.
   - nginx access log에 `500` 응답 5건이 찍혔다.
   - 같은 시각대 앱 로그에 `[load_test] Gemini Stub 실패 시뮬레이션` 과 `비동기 이미지 생성 작업 중 오류 발생`이 같이 남아 있었다.
4. 즉 이번 실패율 `3.03%`은 스레드풀 확장으로 새롭게 생긴 병목이라기보다, 테스트용 실패 주입이 일부 반영된 결과로 해석하는 편이 맞다.
5. 종료 직전 nginx의 `499` 몇 건은 테스트 종료 시점의 client cancel 흔적으로 보이며, summary 집계 판단에서는 분리해서 보는 게 맞다.

## 7. 메트릭 기반 판단
1. 이번 런에서 가장 큰 개선은 CPU 증설이 아니라 **비동기 작업 대기열 감소**다.
2. 종료 직후 CPU는 매우 한가했으므로, 이번 구간의 주병목은 CPU 계산량이라기보다 `60초 외부 I/O 대기 + 큐 적체` 쪽에 더 가깝다.
3. 메모리는 넉넉하다고 보긴 어렵다. 호스트 총 메모리가 1GiB 미만이라 `available 113MiB`, `swap 216MiB 사용` 상태는 여유가 큰 편이 아니다.
4. 따라서 이번 결과만 보면 `core/max` 확장은 유효했지만, 다음 단계에서 더 큰 풀을 바로 밀어 올리기보다는 **Grafana/Prometheus로 peak 메모리와 executor 메트릭을 같이 보는 방향**이 맞다.

## 8. 결론
1. 이번 `2-1` 테스트는 `1-2` 대비 **큰 개선이 있었다**고 판단한다.
2. 특히 `p95`와 `max`가 절반 가까이 줄었고, 처리량도 50% 이상 늘었다.
3. 실패 5건은 서버 자원 고갈의 직접 신호가 아니라 Stub 실패 시뮬레이션 영향이 더 크다.
4. 남은 불확실성은 `실시간 peak CPU/RAM/executor queue`를 못 봤다는 점이다.
5. 따라서 다음 라운드부터는 Prometheus + Grafana를 붙여 executor, JVM, Hikari, 시스템 메트릭을 함께 수집하는 것이 맞다.
