## 4. 해석
1. `10 VU`까지는 안정적이던 구조가 `25~30 VU` 구간에서 급격히 무너졌다.
2. 처리량은 늘었다. 하지만 그 증가를 감당하는 과정에서 실패율이 `74.60%`까지 치솟아, 서비스 관점에서는 이미 임계점을 넘었다.
3. 평균 응답시간은 `60초`대에 머무는 것처럼 보이지만, tail latency가 크게 악화됐다. `p95`가 `65.82s -> 132.05s`로 두 배 가까이 상승했다.
4. 즉 이번 실험은 “가상 스레드가 30 VU까지 안전하다”를 보여준 게 아니라, “가상 스레드를 써도 현재 구조는 25 VU 이상에서 다른 자원 병목으로 무너진다”를 보여준다.

## 5. 실패의 성격
1. k6 기준 실패 요청은 `185건`이다.
2. 앱 로그에서는 동일 구간에 `CannotCreateTransactionException`과 `HikariPool-1 - Connection is not available, request timed out after 30000ms.`가 다수 확인됐다.
3. 즉 이번 실패는 Stub 실패 시뮬레이션이 아니라, DB 커넥션 획득 실패로 인한 `500`이 주된 원인이다.
4. Prometheus의 `http_server_requests_seconds_count` 기준으로도 `/api/v3/generated-images/generate/gemini`에서 `500` 응답 비중이 `200` 응답보다 더 높게 관측됐다.

## 7. 메트릭 기반 판단
1. 이번 런의 1차 병목은 executor가 아니다. active 최대가 `10`, permits 최소가 `6`, reject 증가가 `0`이었다.
2. CPU도 아직 절대 병목 수준은 아니다. 최대 `41.19%`라서 계산량 부족으로 무너졌다고 보기 어렵다.
3. 반면 Hikari pending이 `30`까지 치솟았고, 실제 로그에서도 `Connection is not available, request timed out after 30000ms`가 반복됐다.
4. 즉 `25~30 VU` 구간에서는 가상 스레드가 I/O 대기 비용을 줄여도, DB 커넥션 풀과 트랜잭션 진입 구간이 먼저 한계에 도달한다.
5. 이번 결과는 “가상 스레드만으로 동시성 문제를 끝낼 수 없다”는 증거다. 다음 단계에서는 Hikari 풀 크기, DB 세션 여유, 트랜잭션 범위, 생성 전/후 DB 접근량을 같이 봐야 한다.

## 8. 결론
1. `2-2`까지는 가상 스레드가 tail latency를 줄이고 실패율을 낮추는 방향으로 작동했다.
2. 하지만 `2-3`에서 동시성을 `30 VU`까지 올리자, 주병목은 executor가 아니라 DB 커넥션 풀이 되었다.
3. 따라서 현재 구조의 실질 안정 구간은 `10 VU` 근처이고, `25~30 VU`는 명확한 초과 부하다.
4. 다음 실험 우선순위는 가상 스레드 추가 최적화가 아니다. 먼저 `Hikari / DB connection pool` 여유를 재설정하고, 생성 API 내부의 DB 접근 및 트랜잭션 구간을 줄여야 한다.
